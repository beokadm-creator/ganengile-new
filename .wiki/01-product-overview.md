# 제품/서비스 개요 (Product Overview)

## 한 줄 정의
가는길에는 **요청자(Requester)** 와 **길러(Giller)** 를 매칭해 **배달/수거/출장 형태의 요청을 수행**하게 하는 플랫폼이다.

> 참고: 용어/온보딩 원칙은 `docs/USER-ENTRY-FLOW-STANDARD.md`가 현재 표준 문서다.  
> - [`../docs/USER-ENTRY-FLOW-STANDARD.md`](../docs/USER-ENTRY-FLOW-STANDARD.md)

## 문제 정의(가설)
- 사용자는 “지금 당장/정해진 시간에” 필요한 업무(배달/수거/현장 수행)를 빠르게 요청하고 싶다.
- 수행자는 “전체 배송”이 아니라 **자신이 가능한 구간/미션** 중심으로 참여하고 싶다.
- 운영자는 분쟁/정산/신원/요금 같은 **민감 구간을 통제**할 수 있어야 한다.

> 위 가설은 beta1 공통 배경에 암시돼 있으며, 상세 정책/숫자는 계속 변할 수 있다.  
> - [`../docs/_shared/beta1-core-context.md`](../docs/_shared/beta1-core-context.md)

## 핵심 사용자와 역할
- **요청자(Requester)**: 요청을 만들고 결제/진행상태를 확인한다.
- **길러(Giller)**: 요청을 수락하고 수행(픽업/전달/락커 등) 후 정산을 받는다.
- **운영자(Admin)**: 관리자 웹에서 분쟁/정산/요금정책/연동 설정/심사 등을 관리한다.

## 사용자 여정(요약)

### 요청자 흐름(대표)
1. 계정 생성/로그인
2. 첫 요청 직전에 “이용자 온보딩” 완료(최소 정보)
3. 요청 생성(픽업/전달 위치/시간/물품 등)
4. 매칭 결과 확인 → 진행 추적 → 완료 확인/리뷰

### 길러 흐름(대표)
1. 계정 생성(요청자로도 이용 가능)
2. “길러 신청/전환” 진입
3. 본인확인/정산 정보 등 추가 입력 + 운영 심사(가능)
4. 요청 수락/수행 → 수익/포인트/출금

## 제공 가치(코드에서 확인되는 범위)
- **요청 생성/견적**: 단계형 요청 생성 흐름(`create-request/*`)과 요금 산정 로직
- **매칭/수행**: 길러 라우트 기반 매칭 + 수락/픽업/완료/추적
- **락커 연동**: QR/락커 선택/예약, 외부 락커 API 또는 KRIC 락커 데이터
- **정산/지갑**: 포인트/지갑/출금 + 세금(원천징수) 고려 로직
- **운영**: 관리자 웹에서 분쟁/정산/보증금/요금정책/연동 설정 등
