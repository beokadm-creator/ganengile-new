# 가는길에(ganengile) — 전체 사용자 로직 & 오케스트레이션 미션 보드 설계서 (Figma 이관용)

버전: v0.2  
대상: 고객(배송 요청자) / 길러(배송 수행자) / 전문 배송업체·운영(오케스트레이션)  
서비스 특징: **지하철 동선 기반 생활 배송**, 운영 형태는 **혼합(역-역 인계형 + 문-문(지하철 활용))**

---

## 1) 목적(이 문서가 해결하는 것)
1. 현재 구현된(또는 구현 예정인) “배송 요청 → 배차/수락 → 픽업 → 이동(지하철 포함) → 전달 → 정산/클레임”을 **단일 상태머신**으로 정의합니다.
2. 세 역할(고객/길러/운영)의 화면을 **미션 보드(Workboard)** 형태로 구성할 때 필요한:
   - 컬럼(상태) 정의
   - 카드(업무 단위) 필드 정의
   - 권한/행동(액션) 매트릭스
   - 알림/예외(Incident) 처리 로직  
   을 Figma에 바로 옮길 수 있게 정리합니다.

---

## 2) 핵심 개념(도메인 모델)
- **배송요청(Shipment/Request)**: 고객이 생성하는 최상위 단위(요금/결제/상태의 기준)
- **미션(Mission/Job)**: 수행자(길러 또는 전문업체)에게 할당되는 실행 단위(픽업~드롭 또는 구간)
- **스톱(Stop)**: 픽업지/인계역/배송지 등 방문 지점(혼합 모델 지원)
- **구간(Leg)**: “문→역”, “역→역(지하철)”, “역→문” 같은 이동/인계 단위
- **증빙(Proof)**: 사진/서명/OTP(1회 코드)/QR/위치·시간 로그
- **이슈(Incident)**: 지연, 수취인 부재, 주소 오류, 파손/분실, 인계 실패 등 예외 이벤트(티켓화)

---

## 3) 역할 정의(3자 관점)
### 3.1 고객(배송 요청자)
- 목표: 빠르게 요청 생성, 진행 추적, 커뮤니케이션, 완료 확인, 필요 시 취소/클레임

### 3.2 길러(배송 수행자)
- 목표: 가능한 미션을 받아 효율적으로 수행, 증빙 업로드, 예외 시 운영과 협업, 수익/정산 확인
- 특이점: **길러 신청은 별도 절차**(서비스 문구 기준). 초기에는 고객 계정 중심으로 시작하고, “길러 전환 시점”에 본인확인/정산정보 등록.
- 정의(확정): **전문 길러 = 라이더/기사(현장 배송 수행자)** 입니다.

### 3.3 운영(전문 배송업체/오케스트레이션)
- 목표: SLA(마감/품질) 충족, 배차 최적화(내부 길러 vs 외부 파트너), 실시간 관제, 이슈 대응, 정산/리포트

---

## 4) 전체 상태 머신(배송요청 기준, 권장)
> 보드 컬럼은 이 상태를 그대로 쓰거나, UX에 맞춰 그룹핑(예: 진행중/문제발생)합니다.

### 4.1 정상 플로우 상태
1. **Draft**: 요청 작성 중(임시저장)
2. **Requested**: 요청 생성 완료(결제 전/후 정책에 따라 분기)
3. **PaymentPending / Paid**: 결제 대기 / 결제 완료
4. **DispatchPending**: 배차 대기(길러/파트너 탐색)
5. **Assigned**: 담당(길러/업체) 배정됨
6. **Accepted**: 담당자가 수락
7. **PickupInProgress / PickedUp**: 픽업 진행 / 픽업 완료
8. **InTransit**: 이동 중(지하철 구간 포함)
9. **Delivered**: 전달 완료(+증빙)
10. **Closed**: 정산/CS 포함 최종 클로즈

### 4.2 종료/실패 상태
- **Cancelled**: 취소(고객/운영). 시점에 따라 취소 수수료/환불 규칙 적용
- **Failed**: 배송 실패(증빙/사유 필수)
- **ReturnInProgress / Returned**: 반송 진행/완료(픽업 후 취소/수취불가 등)

---

## 5) 혼합(역-역 + 문-문) 모델링 가이드
### 5.1 요청 타입(카드의 Type/Badge)
- **DoorToDoor**(문-문): 픽업/배송 모두 지상 주소
- **StationToStation**(역-역): 출발역/도착역(또는 역 주변 픽업존)
- **Hybrid**(혼합): 문→역 / 역→역 / 역→문 구간이 결합된 형태

### 5.2 구간(Leg)과 스톱(Stop) 표현
- 하나의 배송요청(Shipment)은 1개 이상의 Leg를 가질 수 있음
- Leg마다:
  - 책임 주체(길러 A / 길러 B / 전문업체)
  - 이동 수단(지하철/도보/기타)
  - 인계 방식(직접 전달/보관함/운영 승인 인계)
  - 증빙 요구(사진/OTP/QR)
  를 지정 가능

---

## 6) 고객(배송 요청자) — 미션 보드 설계
### 6.1 보드 컬럼(추천)
- **작성중**(Draft)
- **결제/확정**(PaymentPending)
- **배차 대기**(DispatchPending)
- **진행중**(Assigned~InTransit)
- **완료**(Delivered/Closed)
- **문제 발생**(Incident Open)
- **취소/실패/반송**(Cancelled/Failed/Returned)

### 6.2 카드(배송요청) 기본 필드
- 요약: 요청번호, 타입(D2D/S2S/Hybrid), 생성시각
- 장소: 픽업/도착(주소 또는 역), 주요 랜드마크
- 시간: 픽업 마감, 배송 마감, 예약 시간(있다면)
- 물품: 크기/무게/취급주의/가액(보험/보상 기준)
- 담당: 현재 담당 길러/업체, 연락(마스킹), 차량/이동수단(정책 허용)
- 상태: 상태 배지 + 다음 단계(예: “수락 대기 03:12”)
- 결제: 요금/결제상태/환불규칙 링크
- 커뮤니케이션: 채팅 스레드 링크, 공지/메모
- 증빙: 픽업/인계/배송 증빙(사진/OTP 성공 여부)

### 6.3 고객 액션(상태별)
- Draft: 저장/삭제/요청 생성
- PaymentPending: 결제 재시도/취소
- DispatchPending: 취소(정책), 요청 내용 일부 수정(주소 변경은 제한)
- Assigned/Accepted: 메시지/연락, 전달 지시 업데이트(시점 제한)
- InTransit: 추적 보기, 인계 예정 안내 확인
- Delivered: 증빙 보기, 평가/팁, 문제 신고(기간 제한 가능)

---

## 7) 길러(배송 수행자) — 미션 보드 설계
### 7.1 보드 컬럼(추천)
- **오퍼(제안받음)**(Offered)
- **수락/예정**(Accepted / Scheduled)
- **픽업해야 함**(To Pickup)
- **이동중**(In Transit)
- **전달/인계 대기**(Handoff Pending / Dropoff Pending)
- **완료**(Done)
- **이슈/도움요청**(Incidents)
- **정산**(Earnings/Settlements)

### 7.2 길러 카드(미션) 기본 필드
- 미션 타입: 픽업/배송/역-역 인계/혼합 구간
- 예상 이동: 출발 스톱 → 도착 스톱, 거리/시간, 지하철 노선/환승 힌트(가능하다면)
- 마감: 픽업 마감/인계 마감/드롭 마감
- 보상: 기본 보상 + 추가(지연 위험/우천/야간)
- 제약: 무게/부피/파손주의/온도/계단/주차 난이도
- 증빙 요구: 사진/OTP/서명/QR
- 커뮤니케이션: 고객/운영 채팅 버튼

### 7.3 길러 액션(상태별)
- Offered: 수락/거절(사유), 오퍼 타이머
- Accepted: 경로 보기, 출발 준비, “지연 예상” 사전 알림
- To Pickup: 도착 체크인(지오펜스), 픽업 증빙 업로드, 픽업 완료 처리
- In Transit: 현재 구간 표시(문→역/역→역/역→문), 위험 알림(지연)
- Handoff/Dropoff Pending: 인계/전달 증빙(OTP/사진/서명), 실패 사유 등록
- Incidents: 이슈 생성(템플릿), 운영 호출, 재시도/반송 전환 요청

---

## 8) 운영(오케스트레이션) — 미션 보드 설계
### 8.1 운영 보드 레이어(권장: 2단)
**A) 주문(Shipment) 보드**
- **신규 유입**(New/Requested)
- **검증 보류**(Validation Hold: 주소/결제/금지품목/시간창 충돌)
- **배차 설계**(Planning: 묶음/라우팅/파트너 선택)
- **진행 모니터링**(Monitoring)
- **예외 대응**(Exceptions)
- **정산/리포트**(Settlement/Reporting)
- **클로즈**(Closed)

**B) 미션(Job) 보드**
- **미할당**(Unassigned)
- **오퍼중**(Offering)
- **수락됨**(Accepted)
- **지연 위험(픽업)**(Pickup At Risk)
- **지연 위험(배송/인계)**(Delivery At Risk)
- **실패/반송 큐**(Fail/Return Queue)

### 8.2 운영 카드(주문/미션) 핵심 필드
- SLA: 픽업/인계/배송 마감, 지연 위험 점수(ETA drift)
- 라우팅: 요청 타입(D2D/S2S/Hybrid), Leg 구성, 인계 포인트
- 배차: 내부 길러/외부 파트너, 오퍼 횟수/거절률, 재배차 히스토리
- 품질: 증빙 누락 여부, 이슈 티켓 수/상태
- 비용: 예상/확정 비용, 보상 상향(서지), 패널티/보류금

### 8.3 오케스트레이션 규칙(권장 정책)
1. **기본 라우팅 선택**
   - 역-역/혼합에서 지하철 구간이 크면: “지하철 친화 길러” 우선
   - 고가/파손주의/야간: 전문 파트너 강제 또는 추가 증빙 강제
2. **오퍼 실패 시 자동 조치**
   - N회 거절/타임아웃 → 보상 상향(서지) → 권역 확장 → 외부 파트너 전환
3. **지연 위험 탐지**
   - 체크인 미발생, ETA 급증, 환승 실패 추정 → 운영 알림 → 고객 선공지/재배차
4. **예외/반송 전환**
   - 픽업 후 취소/수취불가 → ReturnInProgress로 전환, 책임/비용 규칙 적용

## 9) 권한/수정 가능 범위(요약 매트릭스)
> 실제 정책 확정 후, Figma에서 “상태별 액션 버튼”을 이 표로 통제합니다.

- **주소/역 변경**
  - Draft~DispatchPending: 가능(제한적)
  - Assigned 이후: 운영 승인 필요 또는 수수료/재배차 동반
  - PickedUp 이후: 불가(원칙), 예외는 반송/재배송으로 처리

- **취소**
  - DispatchPending 이전: 고객 취소 허용(수수료 정책)
  - Assigned/Accepted: 고객 취소는 “운영 승인 + 수수료”
  - PickedUp 이후: 취소 대신 “반송 전환”

- **증빙**
  - 길러는 업로드/수정(제한 시간), 운영은 검수/잠금, 고객은 열람

## 10) 알림(Notifications) & 커뮤니케이션
### 10.1 고객 알림
- 배차 시작/배정/수락/픽업/인계 진행/배송 완료
- 지연 예상/경로 변경/재배차
- 이슈 발생 및 조치 결과

### 10.2 길러 알림
- 오퍼 도착/마감 임박/고객 메시지/운영 콜
- 증빙 누락 경고/지오펜스 체크인 안내

### 10.3 운영 알림
- SLA 임박, 오퍼 실패, 체크인 미발생, 이슈 급증, 고가물품 경보

---

## 11) 예외(Incident) 템플릿(권장)
> 이슈는 “사유 선택 + 증빙 첨부 + 다음 조치 선택” 3단으로 통일하면 운영 효율이 올라갑니다.

- 수취인 부재(재시도/운영 승인 보관/반송)
- 주소 불명(고객 연락/운영 개입/재배차)
- 인계 실패(대기/대체 인계/재배차)
- 물품 문제(미준비/포장 불량/파손 의심)
- 사고/분실(즉시 작업 잠금 + 운영/보험 플로우)

---

## 12) Figma 이관 가이드(프레임 구조 제안)
### 12.1 Figma 페이지(권장 IA)
1. **01_Overview**: 서비스 한 장 요약(스윔레인/상태머신)
2. **02_Boards**: 고객/길러/운영 보드(컬럼+카드 컴포넌트)
3. **03_Cards**: 카드 상세(요청 상세/미션 상세/이슈 티켓)
4. **04_States**: 상태 배지/타임라인 컴포넌트
5. **05_Permissions**: 상태별 액션 매트릭스(표)
6. **06_Notifications**: 알림 템플릿(푸시/인앱/문자)

### 12.2 보드 공통 UI 컴포넌트
- 상태 배지(색/아이콘)
- 타이머(마감까지 남은 시간)
- 위험도 배지(At Risk)
- 증빙 체크리스트(필수/선택)
- “다음 액션” 버튼(상태 기반)

### 12.3 “스크린샷 자리만 남기고” 보드 구성 템플릿(권장)
> 사용자가 원한 방식: 실제 화면 캡처가 없어도 **스크린샷 영역은 플레이스홀더(빈 박스)** 로만 두고, 보드/로직을 먼저 완성합니다.

**A) 프레임 규격(추천)**
- 보드 프레임: 1440×1024 (Desktop) 또는 1600×900 (발표용)
- 상단 헤더(고정): 72px
- 본문: 좌측 “보드”, 우측 “스크린샷 플레이스홀더 + 설명”

**B) 레이아웃(2컬럼)**
1) 좌측(약 65%): 미션 보드(컬럼 + 카드)
2) 우측(약 35%): 플레이스홀더 스택
   - `SCREENSHOT PLACEHOLDER — 고객 화면`
   - `SCREENSHOT PLACEHOLDER — 길러 화면`

## 14) 전체 사용자 로직(E2E) — 스윔레인(텍스트 버전)
> Figma에서는 이 섹션을 **01_Overview / E2E Swimlane** 프레임으로 옮기면 됩니다.  
> 스크린샷은 필요 없고, 우측에 플레이스홀더만 배치합니다.

### 14.1 정상 플로우(고객 → 길러 → 운영)
1) 고객: **요청 생성(Draft → Requested)**  
   - 픽업/도착(주소 또는 역), 시간창, 물품/취급주의, 증빙 요구(OTP/사진) 입력
2) 고객: **결제(PaymentPending → Paid)** *(선결제 기준, 후결제면 생략/대체)*  
3) 운영: **검증 & 배차 준비(Paid → DispatchPending)**  
   - 주소/역 유효성, 시간창 충돌, 금지 품목, 고가물품 여부 확인
4) 운영: **오퍼 발행(DispatchPending)**  
   - 내부 길러 풀 우선(정책), 실패 시 보상 상향/권역 확장/외부 파트너 전환
5) 길러: **오퍼 수락(Assigned → Accepted)**  
6) 길러: **픽업(PickupInProgress → PickedUp)**  
   - 체크인(지오펜스) + 픽업 증빙(사진/QR/OTP 중 정책)
7) 길러: **이동(InTransit)**  
   - 혼합 모델일 경우 Leg 진행: 문→역 / 역→역 / 역→문
8) 길러: **전달(Delivered)**  
   - 전달 증빙(사진/서명/OTP) 완료
9) 운영: **클로즈(Closed)**  
   - 정산/패널티/보류금 처리 + 고객 알림/영수증

### 14.2 혼합 모델(역-역 + 문-문)에서의 Leg 예시
- 예시 A (Hybrid): `문(픽업) → 역(인계) → 역(인계) → 문(배송)`
  - Leg1: 문→역(길러 A)
  - Leg2: 역→역(지하철, 길러 A 또는 길러 B)
  - Leg3: 역→문(길러 B)
- 예시 B (StationToStation): `역(출발) → 역(도착)`
  - Leg1: 역→역(지하철, 길러)

## 15) 미션 보드 “컬럼” 정의(Entry/Exit 기준 포함)
> 보드 컬럼은 “보기 좋은 그룹핑”이 아니라, **업무가 넘어가는 기준(Entry/Exit)** 이 있어야 운영이 됩니다.

### 15.1 고객 보드(배송요청 기준)
| 컬럼 | 포함 상태 | Entry(들어오는 조건) | Exit(나가는 조건) |
|---|---|---|---|
| 작성중 | Draft | 고객이 임시 저장 | 요청 생성(Requested) 또는 삭제 |
| 결제/확정 | PaymentPending | 결제 필요 | 결제 완료(Paid) / 결제 실패 후 취소 |
| 배차 대기 | DispatchPending | 결제/요청 확정 | 배정(Assigned) / 취소(Cancelled) |
| 진행중 | Assigned~InTransit | 길러/업체 배정 또는 수행 시작 | Delivered/Failed/Return 전환 |
| 완료 | Delivered/Closed | 전달 완료 | 평가/클레임 기간 종료 후 Closed |
| 문제 발생 | Incident Open | 이슈 티켓 Open | 이슈 해결/환불/재배송 등으로 종료 |
| 취소/실패/반송 | Cancelled/Failed/Returned | 종료 상태 | (종료) |

### 15.2 길러 보드(미션 기준)
| 컬럼 | 의미 | Entry | Exit |
|---|---|---|---|
| 오퍼 | 제안받은 미션 | 운영이 길러에게 Offer | 수락/거절/타임아웃 |
| 수락/예정 | 수행 확정 | 길러 수락(Accepted) | 픽업 시작(To Pickup) 또는 운영 취소 |
| 픽업해야 함 | 픽업 액션 대기 | 픽업지/역 도착 전후 | 픽업 완료(PickedUp) |
| 이동중 | 구간 진행 | PickedUp 이후 | 전달/인계 대기 |
| 전달/인계 대기 | 마지막 액션 | 도착/연락/대기 | Delivered 또는 Failed/Return |
| 완료 | 완료됨 | Delivered | (종료) |
| 이슈 | 도움 필요 | Incident 생성 | 해결/재배차/반송 전환 |
| 정산 | 수익/보류금 | 일/주/월 정산 | 정산 완료 |

### 15.3 운영 보드(주문+미션 2단)
**주문(Shipment) 보드**는 “고객 관점 상태”를, **미션(Job) 보드**는 “수행 관점 상태”를 봅니다.  
운영은 두 보드를 오가며 “지연 위험”을 선제적으로 처리합니다.

## 16) 카드 템플릿(보드에서 쓰는 ‘업무 단위’)
### 16.1 고객 카드(배송요청 카드) — 필수/선택 필드
**필수**
- Request ID, 요청 타입(D2D/S2S/Hybrid), 현재 상태
- 픽업/도착(주소 또는 역), 마감 시간(SLA)
- 결제 상태(선결제면 Paid 여부)
- 물품: 크기/무게(최소 등급이라도)

**선택(있으면 운영 효율 상승)**
- 취급주의/고가/보험, 엘리베이터/계단/주차 난이도
- 증빙 요구(사진/OTP/서명)
- 고객 메모(전달 지시)

### 16.2 길러 카드(미션 카드) — 실행 필드
**필수**
- Mission ID, 담당자(본인), 구간(Leg) 타입
- 출발 스톱/도착 스톱 + 마감
- 증빙 요구(체크리스트)
- “다음 액션” 1개(예: 체크인, 픽업 완료, OTP 확인)

**선택**
- 지하철 노선 힌트/환승 수
- 위험도(At Risk), 서지/추가 보상

### 16.3 운영 카드(오케스트레이션 카드) — 의사결정 필드
- SLA(픽업/인계/배송), 지연 위험 점수
- 오퍼 횟수/거절률, 권역 확장 여부
- 파트너 전환 버튼(내부→외부)
- 재배차/취소/반송 전환 버튼(권한 제한)

## 17) 이벤트/알림 매트릭스(최소 권장)
| 이벤트 | 고객 알림 | 길러 알림 | 운영 알림 |
|---|---|---|---|
| 배정됨(Assigned) | 담당자 배정 안내 | 새 미션 배정 | 배정 성공 로그 |
| 수락됨(Accepted) | 수락 안내 + ETA | 픽업 마감 리마인드 | SLA 타이머 시작 |
| 픽업 완료(PickedUp) | 픽업 완료 안내 | 다음 구간 안내 | 지연 위험 재계산 |
| 지연 위험(At Risk) | “지연 예상” 사전 공지 | 마감 임박 경고 | 관제 경보 + 재배차 후보 |
| 전달 완료(Delivered) | 증빙 포함 완료 안내 | 완료/정산 안내 | 검수/클로즈 큐 이동 |
| 이슈 생성(Incident Open) | 이슈 접수/처리중 | 운영 연결 | 예외 큐 진입 |

## 18) 예외 처리(결정 트리 요약)
### 18.1 수취인 부재
1) 연락 성공 → 대체 전달 합의(정책 허용) → 증빙
2) 연락 실패 → 재시도(1~N회) 또는 운영 승인 후 보관/반송

### 18.2 인계 실패(역-역/혼합에서 핵심)
1) 인계 시간 초과 임박 → 운영이 대체 길러 투입(재배차)
2) 인계 불가 확정 → “반송” 또는 “재배송(새 요청)”으로 분기

### 18.3 주소 오류
1) 배정 전: 고객 수정 허용
2) 배정 후: 운영 승인 + 재배차(비용/시간 재산정)
