# 실서비스 스모크 테스트 (앱/웹)

목표: 배포 후 핵심 동선(요청 생성 → 매칭/수락 → 전달/인증 → 정산)이 실제 환경 변수 기준으로 정상 동작하는지 확인한다.

## 0. 사전 조건

- 앱 환경변수에 아래 값이 반영되어 있어야 함
- `EXPO_PUBLIC_SEOUL_FARE_API_URL=https://apis.data.go.kr/B553766/fare`
- `EXPO_PUBLIC_SEOUL_FARE_SERVICE_KEY=발급된 서비스키`
- 테스트 계정 준비
- 사용자 계정 1개
- 길러 계정 1개 (검증 완료 상태)
- 관리자 웹 접근 가능

## 1. 기본 요금 반영 테스트 (앱)

목적: 요청 생성 시 운임 API가 호출되어 요금이 반영되는지 확인.

절차:
1. 사용자 앱 로그인
2. 출발역/도착역 선택 (예: 서울역 0150 → 시청 0151)
3. 요금/가격 상세 화면 진입
4. 요청 생성 직전 가격 요약 확인

기대 결과:
1. 요금 항목이 노출되고 값이 숫자로 표시된다
2. 요금 API 실패 시에도 앱이 중단되지 않고 기본 요금 로직으로 fallback 된다
3. 생성된 요청의 `feeBreakdown.publicFare`가 숫자로 저장된다

## 2. 길러 수락 및 전달 단계 (앱)

목적: 수락 이후 단계 전환이 끊기지 않는지 확인.

절차:
1. 길러 앱 로그인
2. 방금 생성된 요청을 리스트에서 확인
3. 수락 진행
4. 전달 방식 선택
5. 직접 전달 또는 사물함 전달 중 하나로 진행

기대 결과:
1. 상태가 `ACCEPTED` → `IN_PROGRESS`로 전환된다
2. 사물함 전달 선택 시 사물함 검색/선택 화면이 정상 표시된다
3. 전달 완료 처리 시 `DELIVERED` 또는 `AT_LOCKER`로 저장된다

## 3. 사용자 승인 및 정산 연결 (앱 + 관리자 웹)

목적: 전달 완료 → 사용자 승인 → 정산/보증금 처리까지 연결 확인.

절차:
1. 사용자 앱에서 전달 완료 알림 확인
2. 수령 확인/승인 버튼 실행
3. 관리자 웹에서 정산 로그 확인

기대 결과:
1. 요청 상태가 `DELIVERED`/`AT_LOCKER` → `COMPLETED`로 전환된다
2. `settlements/{requestId}`가 생성된다
3. 보증금 환급/길러 정산 기록이 생성된다

## 4. 관리자 확인 (웹)

목적: 관리자에서 운영 데이터 확인 가능 여부 점검.

절차:
1. 관리자 웹 로그인
2. `정산 내역` 페이지 진입
3. `사물함` 페이지에서 지하철/비지하철 탭 확인

기대 결과:
1. 최근 완료 요청의 정산 로그가 조회된다
2. 사물함 목록이 지하철/비지하철로 분리되어 표시된다

## 5. 실패/예외 케이스 점검

목적: API 실패 또는 데이터 누락 상황에서 앱이 멈추지 않는지 확인.

절차:
1. 운임 API 키를 임시로 비우고 앱 재실행
2. 동일한 요청 생성 흐름 재확인

기대 결과:
1. 가격 계산 실패 시에도 앱이 정상 동작한다
2. 요금 항목이 숨김 처리되거나 0으로 처리된다
