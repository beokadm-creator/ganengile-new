# User Entry Flow Standard

## 목적

이 문서는 `가는길에`의 회원 진입 절차를 표준화한다.

핵심 원칙은 다음과 같다.

- 회원가입은 최대한 가볍게 유지한다.
- 실제 서비스 이용 직전에 필요한 정보만 받는다.
- `요청자 이용 시작`과 `길러 전환`은 서로 다른 절차로 본다.

## 표준 용어

### 1. 계정 생성

`계정 생성`은 로그인 가능한 계정을 만드는 단계다.

이 단계에서는 아래만 받는다.

- 이름
- 이메일 또는 소셜 계정
- 비밀번호
- 최소 필수 약관

이 단계의 목적은 `가입 완료`이지 `서비스 이용 준비 완료`가 아니다.

### 2. 이용자 온보딩

`이용자 온보딩`은 요청자가 첫 배송 요청을 하기 전에 진행하는 최소 준비 단계다.

이 단계에서는 아래를 받는다.

- 요청자 이름 확인
- 연락 가능한 휴대폰 번호
- 요청자 서비스 이용에 필요한 필수 약관

이 단계는 `첫 배송 요청 가능 상태`를 만드는 절차다.

즉, 이용자 온보딩은 회원가입과 다르며 `서비스 첫 사용 직전`에 일어난다.

### 3. 길러 온보딩

`길러 온보딩`이라는 말은 내부적으로 쓰더라도, 제품에서는 `길러 신청`, `길러 전환`, `길러 승급`처럼 역할이 드러나는 표현을 우선 사용한다.

이 단계는 요청자 온보딩과 완전히 분리한다.

이 단계에서는 아래가 추가될 수 있다.

- 본인확인
- 이동 구간 정보
- 정산 계좌
- 장비/수행 가능 정보
- 운영 심사

즉, 길러 절차는 `다른 서비스 역할로 전환하는 심사형 프로세스`다.

## 제품 원칙

### 1. 가입에서 많은 정보를 받지 않는다

가입 단계에서 휴대폰 인증, 본인확인, 계좌 등록, 길러 정보를 함께 요구하지 않는다.

이런 정보는 전환 직전 단계에서만 받는다.

### 2. 첫 요청 직전에만 필요한 정보를 받는다

요청자가 실제로 배송 요청을 만들려고 할 때 아래를 확인한다.

- 이용자 온보딩 완료 여부
- 연락 가능한 휴대폰 인증
- 수령인 정보 및 필요한 동의

### 3. 길러 절차는 별도 진입점으로 둔다

길러가 되려는 사용자는 프로필 또는 전용 진입점에서 별도 절차로 이동한다.

요청자 가입/이용 시작 흐름과 혼합하지 않는다.

## 권장 사용자 흐름

### 요청자 흐름

1. 계정 생성
2. 로그인 유지
3. 첫 배송 요청 진입
4. 이용자 온보딩
5. 휴대폰 인증
6. 배송 요청 생성

### 길러 흐름

1. 계정 생성
2. 일반 요청자로 서비스 이용 가능
3. 프로필에서 길러 신청 진입
4. 본인확인
5. 정산 정보 등록
6. 심사 및 승인
7. 길러 역할 활성화

## 금지 원칙

아래는 다시 섞지 않는다.

- 회원가입 = 서비스 이용 준비 완료
- 회원가입 = 길러 신청
- 이용자 온보딩 = 길러 온보딩
- B2B 파트너 정의 = 기업고객 가입 흐름

## 현재 코드 적용 기준

- `NewSignUpScreen`: 계정 생성
- `BasicInfoOnboarding`: 이용자 온보딩
- `CreateRequestScreen`: 첫 요청 진입 시점의 이용 가능 여부 확인
- `GillerApplyScreen`: 길러 신청 절차
